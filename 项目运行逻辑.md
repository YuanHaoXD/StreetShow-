## 1. 项目定位

StreetShow 是一个**AI虚拟试衣与穿搭顾问平台**。

区别于简单换衣工具，StreetShow 的核心能力是：
- **理解衣服的风格语言**（街头感 / 学院风 / 商务正式 / 度假休闲...）
- **动态生成与衣服气质匹配的场景、动作、光影方案**（而非固定列表随机选取）
- **输出专业级穿搭建议**（搭配推荐、适用场合分析）

技术组合：
- 视觉分析：**DMXAPI · qwen3.5-flash**（OpenAI 兼容接口，多模态理解）
- 图像生成：**DMXAPI · gemini-2.5-flash-image**（Gemini 图像编辑，竖版人像输出）
- 后端框架：**FastAPI**（Python 异步 Web 框架）
- 前端框架：**Next.js 14**（React 元框架）

```
python main.py  或  uvicorn main:app --reload
```

启动时按顺序执行：

```
1. main.py
   └── load_dotenv()              ← 从 .env 文件加载环境变量

2. backend/app.py  create_app()
   ├── setup_logging()            ← 初始化日志格式

   │     若缺失则抛出 RuntimeError，阻止启动
   ├── 注册 CORS 中间件            ← 允许跨域（CORS_ALLOW_ORIGINS 配置）
   ├── 挂载 /assets 静态目录       ← 生成的试衣图片从此路径访问

         /api/process-advanced-async
         /health
```

> `/health` 端点返回 `{"status": "ok", "device": "api"}`，可用于监控检活。

         ▼ （并行执行）
 ┌───────────────────────┬─────────────────────────┐
 │   run_qwen_advice()   │   call_nanobanana_api()  │
 │                       │                          │
 │ 将两张图 base64编码    │ 将两张图 base64编码       │
 │ → Qwen-VL API         │ → Nanobanana API         │
 │                       │                          │
 │ 输入：人物图 + 衣物图  │ 输入：人物图 + 衣物图    │

### 4.3 异步多方案（/api/process-advanced-async）

与 4.2 相同的生成逻辑，但**立即返回，后台并行生成图片**，前端轮询进度。

```
POST /api/process-advanced-async
         │
         ▼
 同步完成：Qwen 风格DNA分析 + plan生成
         │
         ▼
 创建 job（内存存储，返回 job_id）
 立即响应：{ job_id, status: "running", results: [{status: "pending"}, ...] }

         │（后台 asyncio.Task 并行运行）
         ▼
 for each variant（并行）:
   call_nanobanana_api() → 生成图片
   update_result(job_id, variant_id, { image_url, status: "done" })

         │（前端每 2.5 秒轮询）
         ▼
 GET /api/process-advanced/jobs/{job_id}
 → 返回当前所有 variant 的最新状态
 → status: "running" | "completed" | "error"
```

> **优势**：用户看到结果是渐进式出现的（哪张先生成哪张先显示），而不是全部等待。

## 5. 风格DNA系统（核心差异化）

### 为什么需要风格DNA？

**旧版问题**：场景和姿势从固定列表随机选取，一件休闲T恤可能被放在"夜景霓虹街头"，一件正式西装被放在"街头路口"——场景与衣服气质不匹配。

**新版解决方案**：在生成方案之前，先用 Qwen-VL 专门分析衣服，提取其风格DNA，然后将这些信息作为上下文注入方案生成提示词，让模型**根据衣服的真实气质自由创作场景**。

### 处理流程

```
garment_image
     │
     ▼
Qwen-VL API（analyze_garment_style）
提示词：分析这件衣服的风格DNA，只输出JSON

     │
     ▼
{
  "style_keywords": ["街头", "oversize", "酷感"],
  "color_palette": ["黑色", "米白"],
  "occasions": ["日常", "逛街", "音乐节"],
  "vibe": "都市街头感，慵懒中带叛逆",
  "avoid_scenes": ["正式场合", "商务会议"]
}

     │
     ▼（注入 build_plan_prompt）
提示词中追加：
  【衣服风格DNA（已预分析）】
    风格关键词：街头、oversize、酷感
    适用场合：日常、逛街、音乐节
    整体氛围：都市街头感，慵懒中带叛逆
    应避免的场景：正式场合、商务会议
  请让所有 variants 的场景、动作、光影与上述风格DNA高度匹配...

     │
     ▼
Qwen-VL 生成的方案中：
  variant 1: 场景=废旧工厂墙壁涂鸦背景，姿势=倚墙单手插兜，光线=街头路灯侧光
  variant 2: 场景=音乐节人群，姿势=律动行走，光线=舞台彩色灯光
  variant 3: 场景=城市街道，姿势=双手插袋悠闲步行，光线=自然日光
```

### JSON 容错机制

Qwen 输出有时不是标准 JSON，系统采用三级容错：

```
Qwen 原始输出
     │
     ▼ 第一级：直接 json.loads() 解析
     │ 失败 ↓
     ▼ 第二级：提取文本中的 {...} 块再解析
     │ 失败 ↓
     ▼ 第三级：调用 Qwen 文本API（qwen-plus）修复 JSON
     │ 失败 ↓
     ▼ 兜底：使用 build_default_plan()（固定模板方案）
```

## 6. 图像生成服务（DMXAPI · Gemini）

通过 DMXAPI 中转平台调用 `gemini-2.5-flash-image` 模型执行多图融合换衣。

### 调用格式

```
POST https://www.dmxapi.cn/v1beta/models/gemini-2.5-flash-image:generateContent
x-goog-api-key: <DMXAPI_KEY>
Content-Type: application/json

{
  "contents": [{
    "role": "user",
    "parts": [
      { "text": "将衣服换上模特；都市街头感，废旧工厂背景..." },
      { "inline_data": { "mime_type": "image/jpeg", "data": "<衣物图base64>" } },
      { "inline_data": { "mime_type": "image/jpeg", "data": "<人物图base64>" } }
    ]
  }],
  "generationConfig": {
    "responseModalities": ["TEXT", "IMAGE"],
    "imageConfig": { "aspectRatio": "2:3" }
  }
}
```

> 认证头为 `x-goog-api-key`（与视觉分析的 `Authorization` 头不同）。
> 宽高比固定为 `2:3`（832×1248），适合人像竖版展示。

### 响应解析

API 返回的图片以 base64 内嵌在 `candidates[].content.parts[].inlineData.data` 中，系统解码后保存为本地文件（`temp/assets/tryon_{uuid}.jpg`），对外暴露为 `/assets/tryon_{uuid}.jpg` 静态路径。

### 失败降级

若 `DMXAPI_KEY` 未配置或请求失败，自动生成一张**灰色占位图**（768×1024，带绿色边框和 "DMXAPI MOCK" 文字），确保整个接口不会崩溃。响应中会标记 `"nanobanana_used": false` 和具体错误信息。

## 7. 并发控制机制

后端使用 `asyncio.Semaphore` 控制对 Qwen-VL API 和 Nanobanana API 的并发请求数：

```python
# backend/core/state.py
SEM = asyncio.Semaphore(MAX_CONCURRENT)  # 默认值：2
```

所有 API 调用都包裹在 `async with SEM:` 中，保证同时最多只有 `MAX_CONCURRENT` 个请求在处理，防止：
- 超出 API 速率限制（Rate Limit）
- 大量并发请求造成内存或网络资源耗尽

可通过环境变量 `MAX_CONCURRENT` 调整。

## 8. 文件生命周期管理

```
temp/
├── uploads/        临时存储用户上传的图片
│   └── {uuid}/     每次请求独立目录
│       ├── person.jpg
│       └── garment.jpg
│
└── assets/         生成的试衣图片（对外提供访问）
    └── tryon_{uuid}.jpg
```

| 目录 | 创建时机 | 删除时机 |
|------|----------|----------|
| `uploads/{uuid}/` | 每次请求开始时 | 请求响应完成后（后台任务立即删除） |
| `assets/tryon_*.jpg` | Nanobanana 生成图片时 | TTL 过期（默认1小时）或超过文件数量上限（默认200个） |

`cleanup_assets()` 在每次请求完成后的后台任务中执行，按修改时间排序删除过期和超量文件。

## 9. 错误降级策略

整个系统遵循"**永不崩溃，总能返回结果**"的原则：

| 组件 | 错误场景 | 降级处理 |
|------|----------|----------|
| 图片上传 | 非图片文件 / 超过大小限制 | 返回 400 错误 |
| Qwen API | 返回非 JSON / 解析失败 | 三级容错 → 兜底默认 plan |
| Qwen API | 网络超时 / Key 无效 | 抛出异常 → 使用默认 plan |
| Nanobanana | Key 未配置 | 直接返回 mock 图 |
| Nanobanana | 请求失败 / 超时 | 捕获异常 → 返回 mock 图，记录错误 |
| 异步任务 | 单个 variant 失败 | 该 variant 标记 `status: "error"`，其他继续 |
| 风格DNA分析 | 解析失败 | 返回通用兜底风格数据，继续生成方案 |

## 11. 环境配置

所有敏感配置通过 `.env` 文件管理（不提交到 git）。

### 必填项

| 变量名 | 说明 | 获取方式 |
|--------|------|----------|
| `DMXAPI_KEY` | DMXAPI 统一密钥（视觉分析 + 图像生成共用） | www.dmxapi.cn |

### 可选项（均有默认值）

| 变量名 | 默认值 | 说明 |
|--------|--------|------|
| `DMXAPI_VL_MODEL` | `qwen3.5-flash` | 视觉分析模型 |
| `DMXAPI_TIMEOUT` | `90` | API 请求超时时间（秒） |
| `MAX_CONCURRENT` | `2` | 最大并发 API 调用数 |
| `MAX_UPLOAD_MB` | `10` | 上传图片最大体积（MB） |
| `MAX_IMAGE_EDGE` | `2048` | 图片最长边限制（px） |
| `ASSET_TTL_SECONDS` | `3600` | 生成图片的存活时间（秒） |
| `LOG_LEVEL` | `INFO` | 日志级别 |

### 快速启动

```bash
# 1. 配置环境变量
cp .env.example .env
# 编辑 .env，填入 DMXAPI_KEY（一个 Key 即可）

# 2. 安装后端依赖
pip install -r requirements.txt

# 3. 启动后端（开发模式）
uvicorn main:app --host 0.0.0.0 --port 8000 --reload

# 4. 安装并启动前端
cd frontend
npm install
npm run dev
# 浏览器打开 http://localhost:3000
```
